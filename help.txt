### Prompt 2: Multicloud Mapper UI

**Title:** Feature Specification: Multicloud Mapper Interface
**Objective:** Develop a user interface that allows users to select a cloud provider and input a service name to retrieve mapping details via an Agent.

**1. Functional Requirements:**
*   **UI Component - Cloud Provider Selection:**
    *   Must display a dropdown menu.
    *   Dropdown options must strictly include: `Azure`, `AWS`, `GCP`.
*   **UI Component - Service Input:**
    *   Must provide a text input box to capture the cloud service name (e.g., "S3", "Blob Storage").
*   **Action - Search:**
    *   Must include a "Search Service" button.
    *   Upon click, the button must trigger the backend Mapping Agent with the selected provider and service name.
*   **Output - Results Display:**
    *   Must display the Agent's response in a tabular format (Grid).
    *   The table must include columns for: `Service Name`, `Category`, `Description`, and `Mapping Status`.

**2. Technical Specifications:**
*   **Frontend:** React/Angular/Vue (select based on stack).
*   **State Management:** Handle loading states (spinner) while the Agent is processing.
*   **Error Handling:** Display user-friendly error messages if the Agent returns no data or fails.

**3. Acceptance Criteria:**
*   **Given** the user selects "AWS" and types "Lambda", **When** they click "Search Service", **Then** the system triggers the Agent and displays the equivalent services in the results table.
*   **Given** the user clicks search without entering a service name, **Then** the system displays a validation error "Service name is required."

### Prompt 3: Azure OpenAI Architecture Validation & BOM Agent

**Title:** Backend Specification: Architecture Diagram Validation & BOM Agent
**Objective:** Create an Azure OpenAI-based agent capable of validating uploaded architecture diagrams and generating a Bill of Materials (BOM) with cost estimates.

**1. Functional Requirements:**
*   **Input:** Accept image file uploads (PNG, JPG, JPEG) of architecture diagrams.
*   **Step 1 - Validation Logic:**
    *   The Agent must first analyze the image to determine if it is a valid technical architecture diagram.
    *   **If Invalid:** Return a response indicating `isValid: false` and stop processing.
    *   **If Valid:** Proceed to Step 2.
*   **Step 2 - Analysis & BOM Generation:**
    *   Identify all components within the diagram.
    *   Generate a Bill of Materials (BOM) list.
    *   Estimate the cost for each component (utilizing GPT knowledge or integrated pricing APIs).
*   **Output Format:** Return a structured JSON object containing:
    *   `isValid` (Boolean)
    *   `bomList` (Array of Objects: `{ ComponentName, Type, EstimatedCost, Description }`).

**2. Technical Specifications:**
*   **AI Model:** Azure OpenAI GPT-4 (Vision capable model).
*   **Prompt Engineering:** Implement system prompts to enforce strict JSON output structure.
*   **Security:** Ensure API keys are stored in Key Vault/Environment Variables.

**3. Acceptance Criteria:**
*   **Given** a user uploads a photo of a landscape, **When** the Agent processes it, **Then** the Agent returns `isValid: false`.
*   **Given** a user uploads a valid AWS architecture diagram, **When** the Agent processes it, **Then** the Agent returns `isValid: true` and a JSON list of components with estimated costs.

### Prompt 5: Cross-Cloud BOM Suggestion Logic

**Title:** Feature Specification: Non-Azure to Azure BOM Mapping
**Objective:** Enhance the BOM Generator logic to detect non-Azure technologies and automatically generate an equivalent Azure Bill of Materials.

**1. Functional Requirements:**
*   **Trigger:** This logic activates during the BOM generation process (Prompt 3/4).
*   **Detection:** The Agent must identify if the uploaded diagram uses non-Azure services (e.g., AWS, GCP, On-premise).
*   **Logic - Azure Mapping:**
    *   If non-Azure services are detected, the Agent must map each service to its Azure equivalent (e.g., AWS S3 -> Azure Blob Storage).
*   **Output Extension:**
    *   The result JSON must include an additional field: `AzureEquivalentBOM`.
    *   The UI Table (from Prompt 4) must render an additional column or a separate section labeled "Azure Equivalent Services".

**2. Technical Specifications:**
*   **Prompt Engineering:** Update the GPT system prompt to include instructions: *"If the detected services are not native to Azure, suggest the closest Azure equivalent for each item."*
*   **UI Update:** The BOM table must conditionally render the "Azure Equivalent" column only if the original diagram was non-Azure.

**3. Acceptance Criteria:**
*   **Given** a user uploads an AWS architecture diagram, **When** BOM generation completes, **Then** the UI displays the original AWS components alongside a column suggesting the relevant Azure alternatives.
*   **Given** a user uploads a native Azure architecture diagram, **When** BOM generation completes, **Then** the "Azure Equivalent" column is hidden or duplicates the original component.

### Prompt 6: Multicloud Mapper "Others" & Azure Suggestion

**Title:** Feature Specification: Third-Party Technology Mapping
**Objective:** Extend the Multicloud Mapper to handle generic technologies or third-party products and provide Azure mapping suggestions.

**1. Functional Requirements:**
*   **UI Update - Dropdown:**
    *   Update the Cloud Provider dropdown to include an option: `Others`.
*   **Input Logic:**
    *   When `Others` is selected, the text box placeholder changes to "Enter Technology/Product Name" (e.g., "Kubernetes", "MongoDB").
*   **Agent Logic:**
    *   The Agent must identify the technology type.
    *   The Agent must identify the current hosting cloud (if known) or treat it as agnostic.
    *   **Action:** Generate a mapping to the most relevant Azure service (e.g., Kubernetes -> AKS, MongoDB -> CosmosDB).
*   **Output Display:**
    *   Results must be shown in the existing tabular format.
    *   Table columns: `Input Technology`, `Identified Category`, `Suggested Azure Service`, `Description`.

**2. Technical Specifications:**
*   **Backend:** Reuse existing Mapper Agent infrastructure but branch logic based on `provider == "Others"`.
*   **Response Model:** Ensure the JSON response model supports the new "Suggested Azure Service" field.

**3. Acceptance Criteria:**
*   **Given** the user selects "Others" and types "Kafka", **When** they click Search, **Then** the table displays a row mapping Kafka to "Azure Event Hubs" or "Confluent on Azure".
*   **Given** the user selects "Others" and types a generic term like "Database", **Then** the Agent returns a list of relevant Azure database services (SQL Database, PostgreSQL, etc.).
